## 1. Abstract

We present an intelligent reading comprehension and quiz generation system built on the RACE (ReAding Comprehension from Examinations) dataset. The system combines classical machine learning models for answer verification, question generation, distractor creation, and hint generation into an interactive Streamlit-based web application. Model A handles answer verification using Logistic Regression, SVM, Naive Bayes, Random Forest, XGBoost, and ensemble methods, alongside unsupervised approaches (K-Means, GMM) and semi-supervised learning (Label Propagation). Model B generates plausible distractor options using frequency-based candidate extraction with Random Forest ranking and produces graduated hints via sentence-level cosine similarity scoring. The system processes approximately 87,000 training samples from Chinese middle and high school English exams. Our best supervised model achieves a Macro F1 of 0.45 on the binary verification task, while the Random Forest difficulty estimator achieves 83% accuracy. The distractor generator maintains 100% accuracy in excluding the correct answer, and the hint system achieves 57.7% Precision@K. The complete system is deployed as a four-screen Streamlit application featuring article input, interactive quiz view with color-coded feedback, graduated hint panels, and a developer analytics dashboard with metrics visualization and CSV export capabilities.

## 2. Introduction & Motivation

Reading comprehension is a fundamental skill in education, and automated assessment tools can significantly reduce instructor workload while providing students with immediate, interactive feedback. Traditional assessment systems rely on manually crafted questions and answer keys, which are time-consuming to produce and difficult to scale.

The goal of this project is to build an end-to-end system that:

1. **Verifies answers** to reading comprehension questions using classical ML classifiers.
2. **Generates questions** from reading passages using template-based approaches ranked by ML models.
3. **Creates plausible distractors** (wrong answer options) that are semantically related but incorrect.
4. **Provides graduated hints** that progressively reveal information to guide learners.
5. **Presents everything** in an accessible, interactive Streamlit web interface.

The RACE dataset provides an ideal testbed: it contains ~100,000 multiple-choice questions across two difficulty levels (middle and high school), generated by human English teachers, ensuring high quality and diversity.

### Why Classical ML?

While large language models (LLMs) dominate NLP leaderboards, classical ML approaches remain valuable for:
- **Interpretability**: Feature importances and decision boundaries are transparent.
- **Efficiency**: Models train in seconds on a standard laptop, no GPU required.
- **Educational value**: Understanding TF-IDF, cosine similarity, and ensemble methods is foundational to ML education.

## 3. Related Work

1. **Lai et al. (2017)** — *"RACE: Large-scale ReAding Comprehension Dataset From Examinations."* EMNLP 2017. Introduced the RACE dataset with ~28,000 passages and ~100,000 questions, establishing that a significant gap exists between state-of-the-art models (43% accuracy) and human performance (95%).

2. **Rajpurkar et al. (2016)** — *"SQuAD: 100,000+ Questions for Machine Comprehension of Text."* EMNLP 2016. Introduced the extractive QA paradigm where answers are spans within the passage. While RACE uses multiple-choice format, SQuAD's feature engineering insights (word overlap, attention mechanisms) influenced our approach.

3. **Welbl et al. (2017)** — *"Crowdsourcing Multiple Choice Science Questions."* NeurIPS Workshop. Explored automated distractor generation using knowledge bases and word embeddings, demonstrating that semantic similarity is a key factor in creating plausible distractors.

4. **Gao et al. (2019)** — *"Generating Distractors for Reading Comprehension Questions from Real Examinations."* AAAI 2019. Proposed neural distractor generation models trained on real exam data. Our work adapts the candidate-extraction-and-ranking paradigm using classical ML features.

5. **Mitkov et al. (2006)** — *"A Computer-Aided Environment for Generating Multiple-Choice Test Items."* Natural Language Engineering. Pioneered template-based question generation with NLP, combining syntactic parsing with semantic similarity for question ranking — an approach we adapt using TF-IDF features.

6. **Pan et al. (2020)** — *"Unsupervised Multi-hop Question Answering by Question Generation."* NeurIPS 2020. Demonstrated that question generation and answering can be jointly optimized. Our template-based approach is a simplified version suited for classical ML pipelines.

## 4. Dataset Analysis

### 4.1 Overview

The RACE dataset contains reading comprehension questions collected from Chinese middle and high school English exams.

| Split | Samples | Unique Articles |
|-------|---------|-----------------|
| Train | 87,866 | ~25,000 |
| Validation | 4,887 | ~1,400 |
| Test | 4,934 | ~1,400 |

Each sample consists of: `id`, `article`, `question`, and four options (`A`, `B`, `C`, `D`) with a correct answer label.

### 4.2 Answer Distribution

The answer labels are slightly imbalanced:
- **A**: 19,146 (21.8%)
- **B**: 22,726 (25.9%)
- **C**: 23,891 (27.2%)
- **D**: 22,103 (25.2%)

Class balance ratio (max/min): 1.25, indicating mild imbalance.

### 4.3 Article Length

- Average word count: ~250 words per article
- High school articles are significantly longer than middle school articles
- Sentence count ranges from 3 to 40+ per article

### 4.4 Question Types

Analysis of question prefixes reveals:
- **Fill-in-the-blank**: Most common (cloze-style questions with blanks)
- **What/Which/Who/Where/When/Why/How**: Standard Wh-question types
- The distribution skews heavily toward fill-in-the-blank and "What" questions

### 4.5 Difficulty Levels

Questions are sourced from two levels:
- **High School**: ~70% of dataset, longer articles, more complex reasoning
- **Middle School**: ~30%, shorter articles, more factual questions

## 5. Model A: Design, Training, Results

### 5.1 Feature Engineering

For the answer verification task, each sample is expanded into 4 binary classification instances (one per option). Features include:

| Feature | Description |
|---------|-------------|
| `tfidf_cos_art_opt` | TF-IDF cosine similarity between article and option |
| `tfidf_cos_q_opt` | TF-IDF cosine similarity between question and option |
| `tfidf_cos_art_q` | TF-IDF cosine similarity between article and question |
| `oh_cos_art_opt` | One-Hot cosine similarity between article and option |
| `oh_cos_q_opt` | One-Hot cosine similarity between question and option |
| `option_word_count` | Number of words in the option |
| `question_word_count` | Number of words in the question |
| `article_word_count` | Number of words in the article |
| `overlap_art_opt` | Word overlap ratio between article and option |
| `overlap_q_opt` | Word overlap ratio between question and option |
| `overlap_art_q` | Word overlap ratio between article and question |

The TF-IDF vectorizer was fitted on training articles with 10,000 features, bigrams, and sublinear TF scaling. The One-Hot (CountVectorizer) uses binary encoding.

### 5.2 Supervised Models

| Model | Accuracy | Macro F1 | Precision | Recall | Train Time |
|-------|----------|----------|-----------|--------|------------|
| Logistic Regression | 0.750 | 0.429 | 0.375 | 0.500 | 0.01s |
| SVM (LinearSVC) | 0.750 | 0.429 | 0.375 | 0.500 | 0.02s |
| XGBoost | 0.745 | 0.445 | 0.538 | 0.503 | 0.23s |
| Naive Bayes (Question Type) | 0.647 | 0.329 | 0.404 | 0.317 | 0.01s |
| Random Forest (Difficulty) | 0.830 | 0.787 | 0.800 | 0.777 | 0.83s |

The difficulty estimation task (middle vs. high school) is best served by Random Forest, leveraging article length and sentence count as key discriminative features.

### 5.3 Unsupervised & Semi-Supervised Models

**Label Propagation** (semi-supervised with 10% labeled data): Achieved Macro F1 of 0.463 on validation, demonstrating that semi-supervised approaches can approach supervised baselines with limited labeled data.

### 5.4 Ensemble Methods

| Ensemble | Accuracy | Macro F1 |
|----------|----------|----------|
| Soft Voting (LR + RF + XGBoost) | 0.750 | 0.429 |

### 5.5 Template-Based Question Generation

The system generates questions via:
1. **Candidate sentence extraction**: Sentences ranked by keyword overlap with the correct answer.
2. **Wh-word templates**: Applied to top sentences (What/Who/Where/When/Why/Which).
3. **ML ranking**: Generated questions can be ranked by SVM or Random Forest confidence.

## 6. Model B: Design, Training, Results

### 6.1 Distractor Generation

**Architecture:**
1. Extract candidate phrases from the article (unigrams, bigrams, trigrams) excluding sentences containing the correct answer.
2. Compute features for each candidate: character match score, length ratio, passage frequency, word overlap, TF-IDF and One-Hot cosine similarities to the correct answer.
3. Rank candidates using a Random Forest classifier trained on labeled distractor data.
4. Apply diversity penalty to ensure variation among selected distractors.

**Results:**
- Distractor Ranker F1: 0.558
- Distractor Accuracy (correct answer excluded): **100%**
- Top features: `len_ratio` (0.464), `char_match` (0.331), `passage_freq` (0.166)

### 6.2 Hint Generation

**Architecture:**
1. Split article into sentences.
2. Score each sentence using: keyword overlap with question (40%), keyword overlap with answer (40%), TF-IDF cosine similarity (20%).
3. Select 3 graduated hints: general (least relevant in top half), medium, near-explicit (most relevant).

**Results:**
- Hint Scorer Accuracy: 99.9%
- Hint Precision@K: **57.7%** (hints containing answer keywords)

### 6.3 Word2Vec Distractors (Optional)

Implemented but not run by default (requires 1.5GB download). Uses `word2vec-google-news-300` to find nearest neighbours of answer words as alternative distractor candidates.

## 7. User Interface Description

The Streamlit application consists of four screens:

### Screen 1 — Article Input
- Text area for pasting custom reading passages
- "Load Random Sample" button fetches from the RACE test set
- "Submit & Generate Quiz" triggers Model A and Model B inference

### Screen 2 — Quiz View
- Displays the question with 4 radio-button options
- "Check Answer" button provides color-coded feedback (green for correct, red for incorrect)
- Expandable section shows Model A confidence scores for all options
- "New Question" button loads a fresh sample

### Screen 3 — Hint Panel
- Three graduated hints revealed one at a time
- Hint 1: General clue (least relevant sentence)
- Hint 2: More specific (medium relevance)
- Hint 3: Near-explicit (most relevant sentence)
- "Reveal Answer" button appears after all hints are shown

### Screen 4 — Developer Dashboard
- **Model A tab**: Bar charts comparing all models (accuracy, F1, precision, recall), metrics table, best model summary
- **Model B tab**: Distractor and hint performance metrics
- **Session Analytics tab**: Questions answered, accuracy, latency graph
- **Export tab**: CSV download of session results

## 8. Evaluation & Discussion

### 8.1 Key Findings

1. **Answer verification is hard**: The 75% majority baseline (always predict "incorrect") is difficult to beat with lexical features alone. XGBoost shows the most promise with Macro F1 of 0.445.

2. **Difficulty estimation works well**: Article length and sentence count are strong signals for middle vs. high school classification (83% accuracy).

3. **Distractor generation is effective**: The system never includes the correct answer among distractors, and the diversity penalty ensures varied options.

4. **Hints are moderately useful**: 57.7% of generated hints contain at least one answer keyword, providing genuine guidance to learners.

### 8.2 Performance Bottlenecks

- Feature engineering (TF-IDF transform per option) is the main latency contributor.
- Full dataset training requires optimization (batch TF-IDF transforms instead of per-sample).

---

## 9. Limitations & Future Work

### Limitations
1. **Lexical features are insufficient** for complex reasoning questions that require inference beyond word overlap.
2. **Template-based question generation** produces generic questions; neural generation (T5, GPT) would be more natural.
3. **Binary verification framing** (correct/incorrect per option) doesn't capture inter-option relationships.
4. **Training on subset**: Current models are trained on 5,000 rows for speed; full-dataset training would improve performance.

### Future Work
1. **Transformer-based features**: Use BERT/RoBERTa embeddings as features for classical ML models (no fine-tuning needed).
2. **Neural question generation**: Fine-tune T5 or GPT-2 for question generation.
3. **Multi-class formulation**: Predict the correct answer directly (4-class) instead of binary verification.
4. **Active learning**: Use Label Propagation to iteratively expand labeled data.
5. **Adaptive difficulty**: Adjust question difficulty based on user performance in the UI.

---

## 10. Conclusion

We built a complete reading comprehension quiz system combining classical ML for answer verification, distractor generation, and hint scoring with an interactive Streamlit UI. The system demonstrates that while classical approaches have limitations on complex reasoning tasks, they provide fast, interpretable, and resource-efficient solutions suitable for educational applications. The modular architecture (preprocessing → Model A → Model B → UI) supports easy extension with more advanced models in the future.
